# Liso Server 请求解析与回显封装实现记录

## 1. 需求目标

需要在当前项目中实现并验证以下行为：

1. Server 收到 Client 发送的、包含 HTTP 请求行的消息后，能够正确解析。
2. 如果请求方法是 `GET`、`HEAD`、`POST`，Server 将解析结果重新封装（encapsulation）为 HTTP 请求报文并回传给 Client。
3. 其他情况给出合理错误响应（例如解析失败、方法不允许）。
4. 在 Docker 环境中完成编译与联调验证。

---

## 2. 修改文件清单

本次实现涉及 3 个核心文件：

1. `src/echo_server.c`
2. `src/echo_client.c`
3. `Makefile`

## 3. 关键问题与设计思路

### 3.1 旧版 `echo_server` 的问题

初始框架中的 `echo_server` 逻辑是“收到字符串 -> 拼接 `(echo back)` -> 发回”，并没有：

1. 调用 HTTP 解析器；
2. 按方法（GET/HEAD/POST）进行语义处理；
3. 进行请求级别的“重封装回显”。

因此它无法满足实验要求中的“请求行解析 + 按方法封装回传”。

### 3.2 旧版 `echo_client` 的问题

初始 `echo_client` 使用 `fgets()`，只读取标准输入的第一行。
这会导致：

1. 发给 Server 的内容仅有请求行（例如 `GET / HTTP/1.1`）；
2. 请求头和 `\r\n\r\n` 结束标记丢失；
3. 解析器无法识别完整 HTTP 头部，触发 400。

所以必须改成“从 stdin 读取并发送完整报文”。

### 3.3 编译链接问题

`echo_server` 在接入 `parse()` 后，链接阶段需要解析器相关目标文件：

1. `parse.o`
2. `y.tab.o`
3. `lex.yy.o`

否则会出现未定义符号。

## 4. 具体实现

### 4.1 `src/echo_server.c`：接入解析和封装回传

主要改动如下：

1. 引入 `parse.h`，以调用 `parse()` 并访问 `Request` 结构。
2. 新增 `build_encapsulated_request()`：
   1. 将 `Request` 中的 method/uri/version 写回首行；
   2. 依次写回所有 header；
   3. 追加 `\r\n` 结束头部；
   4. 进行长度边界检查，防止缓冲区溢出。
3. 在收包后改为：
   1. `parse(buf, readret, client_sock)` 解析请求；
   2. 解析失败返回 `400 Bad Request`；
   3. 方法为 GET/HEAD/POST 时，按解析结构重封装并 `send()`；
   4. 其他方法返回 `405 Method Not Allowed`；
   5. 发送后释放 `request->headers` 和 `request`。

### 4.2 `src/echo_client.c`：发送完整请求报文

主要改动如下：

1. 将 `fgets()` 改为 `fread(msg, 1, BUF_SIZE - 1, stdin)`；
2. 根据实际读取长度 `total_to_send` 发送，不再使用 `strlen(msg)`；
3. 保证从管道输入样例文件时，请求头和结束符会完整送达 Server。

### 4.3 `Makefile`：补齐 `echo_server` 链接依赖

将 `echo_server` 目标从：

- 仅依赖 `echo_server.o`

改为依赖：

1. `echo_server.o`
2. `parse.o`
3. `y.tab.o`
4. `lex.yy.o`

确保 `parse()` 及词法/语法解析符号可正确链接。

## 5. 功能验证过程（Docker）

### 5.1 镜像构建

```bash
docker build -t liso-lab:latest -f ./Dockerfile .
```

### 5.2 容器内重编译并联调

```bash
docker run --rm -v /home/luchitong/computer-network-lab:/home/project-1 liso-lab:latest /bin/bash -lc '
set -e
cd /home/project-1
make clean
make
./echo_server >/tmp/server.log 2>&1 & SERVER_PID=$!
sleep 1
echo "===== GET ====="
cat samples/request_get | ./echo_client 127.0.0.1 9999
echo "===== HEAD ====="
cat samples/request_head | ./echo_client 127.0.0.1 9999
echo "===== POST ====="
cat samples/request_post | ./echo_client 127.0.0.1 9999
kill $SERVER_PID
wait $SERVER_PID || true
echo "===== SERVER LOG (tail) ====="
tail -n 80 /tmp/server.log
'
```

### 5.3 观测结果

验证中可见：

1. GET/HEAD/POST 均能被 Server 解析；
2. Client 收到的是“按解析结构重新封装”的请求报文回显；
3. Server 日志包含 `parsing_request: Matched Success.` 和 `Send encapsulated response for method: <METHOD>`。

结论：满足需求。

## 6. 典型排查记录

### 6.1 400 Bad Request 的常见原因

如果手输 `GET / HTTP/1.1\r\n\r\n`，很多终端会把它当作字面字符发送，而不是真实 CRLF 字节。
解析器需要真实的 `\r\n\r\n` 才能识别头结束。

建议使用：

```bash
printf 'GET / HTTP/1.1\r\nHost: localhost\r\n\r\n' | ./echo_client 127.0.0.1 9999
```

或直接使用样例：

```bash
cat samples/request_get | ./echo_client 127.0.0.1 9999
```

### 6.2 GLIBC 版本不匹配

若把宿主机编译出的二进制直接放到容器内运行，可能出现 `GLIBC_xxx not found`。
解决方式是在容器内重新 `make clean && make`。
