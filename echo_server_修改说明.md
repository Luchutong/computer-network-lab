# Echo Server 修改说明（Week1）

## 1. 修改背景

自动评测结果显示服务端在以下场景出现超时：

1. good request: timeout reached
2. bad request: timeout reached
3. uni request: timeout reached

这说明原始实现虽然能接收并回显部分内容，但没有满足作业要求的“按请求逐条解析并响应”的行为，特别是在一条连接中携带多个请求（pipeline）时，无法完整处理。

---

## 2. 为什么要改

根据题目要求，服务器必须区分并处理三类情况：

1. Echo：当方法为 GET / HEAD / POST 时，解析后重新封装并回传。
2. 未实现：当方法不是 GET / HEAD / POST 时，返回：
   HTTP/1.1 501 Not Implemented\r\n\r\n
3. 格式错误：当请求格式不合法时，返回：
   HTTP/1.1 400 Bad Request\r\n\r\n
原版本主要问题：

1. 只按一次 recv 的内容做单次处理，没有在连接内循环切分“多个完整请求”。
2. 遇到 pipeline 场景时，后续请求可能被遗漏，导致客户端等待超时。
3. 未实现方法和格式错误的分支不完全符合本次实验要求。

---

## 3. 改了什么

本次只修改了：

- src/echo_server.c

关键改动如下：

1. 新增请求边界识别函数 `find_request_end`
   - 在连接缓冲区中查找 \r\n\r\n，定位一个完整 HTTP 请求头结束位置。

2. 新增请求行快速分类函数 `classify_request_line`
   - 先读取请求行（第一行）进行方法分类：
     - GET / HEAD / POST -> 允许进入解析与回显
     - 其他合法方法 -> 直接返回 501
     - 请求行不合法 -> 返回 400

3. 引入连接级缓冲区并循环处理
   - 每次 recv 后把数据追加到连接缓冲区。
   - 通过 while 循环不断切出“一个完整请求”并立即响应。
   - 处理完一个请求后，用 memmove 丢弃已处理字节，继续处理剩余字节。
   - 这样可正确处理一次收包内包含多个请求的情况。

4. 保持 GET/HEAD/POST 的“封装回显”
   - 仍通过已有封装函数 `build_encapsulated_request` 生成回显报文。

5. 关键路径补充中文注释
   - 在请求切分、分类、错误分支、缓冲区处理等关键处增加中文注释，便于后续审计。

---

## 4. 实现思路（流程）

服务端针对单个连接的大致处理流程：

1. recv 读入数据并追加到连接缓冲区。
2. 在缓冲区查找是否存在完整请求（\r\n\r\n）。
3. 如果有完整请求，先做请求行分类：
   - 分类为支持方法：调用 parse，再封装回显。
   - 分类为未实现方法：返回 501。
   - 分类为格式错误：返回 400。
4. 从缓冲区移除已处理请求，继续循环处理后续请求。
5. 若对端关闭连接且缓冲区仍有残缺报文，按 400 处理后结束。

---

## 5. 结果

修复后可满足题目要求：

1. GET/HEAD/POST 可解析并回显（封装后返回）。
2. 非 GET/HEAD/POST 方法返回 501 Not Implemented。
3. 格式错误请求返回 400 Bad Request。
4. 对多请求（pipeline）可逐条响应，避免因遗漏后续请求导致超时。

---

## 6. 额外说明

当前工作区里 obj 目录下部分目标文件可能是 root 权限（来自容器内构建），在宿主机执行 make clean 时会出现 Permission denied。该问题不影响上述逻辑修复本身，但会影响本机直接 clean/rebuild。若需要本机完整重编译，请先修复 obj 目录权限。
